Java Service System — Contact, Task & Appointment Management

A Java management system refactored from static ArrayList storage to HashMap-backed services, with search, sort, and date-range filtering implemented using Java Streams and full JUnit 5 test coverage maintained throughout.

Artifact Description

This artifact is a Java-based management system built in CS-320: Software Testing, Automation, and Quality Assurance. The original project consisted of three service modules — ContactService, TaskService, and AppointmentService — each handling a different type of record with support for adding, retrieving, and deleting entries. Testing was performed in Eclipse with Maven and JUnit 5, validating field constraints and confirming correct behavior for valid and invalid inputs. The original code worked but was limited: no search, sort, or filtering capabilities, and a structural flaw in ContactService that caused tests to interfere with each other.

Why I Selected This Artifact

I chose this artifact because it represents an honest starting point rather than a polished one. The original code had real structural problems — a static shared ArrayList used as instance storage, O(n) lookups on every read/delete, and no ability to query records meaningfully. That made it a better demonstration of growth than something already optimized.

The refactoring process required me to understand not just how to write better code, but why the original design failed at the system level. A static field shared across every class instance is not visible as a bug until tests start corrupting each other — and then it becomes obvious, fast.

Enhancements Implemented

  • Replaced ArrayList storage in all three services with HashMap<String, T>, reducing lookup, insert, and delete operations from O(n) to O(1)
  • Resolved static shared-state bug in ContactService that caused test runs to interfere with each other
  • Implemented keyword search by name (contacts), description (tasks), and description (appointments) using Java Streams
  • Added alphabetical sort by last name for contacts
  • Added appointment search by date and date-range filter returning only upcoming appointments within a configurable window
  • Maintained full JUnit 5 test coverage throughout the refactor — including tests that check actual return values, not just that objects were created
  • Refactored ContactService to separate data record, storage, and service logic into distinct responsibilities

Course Outcomes Addressed

Algorithmic Principles

HashMap selection over ArrayList was a deliberate complexity decision — O(1) vs O(n) — reasoned against the actual access patterns of the service layer.

Well-Founded Techniques

Java Streams for filtering and sorting are idiomatic, readable, and maintainable — the right tool for collection processing in modern Java.

Software Quality

Full JUnit 5 coverage with value-checking assertions (not just creation assertions) enforces correctness and catches regressions across the refactor.

Collaborative Environments

Separating data record, storage, and service logic makes each layer independently understandable and testable by any team member.

Reflection

The most instructive moment of this enhancement was discovering how much the original design resisted testing. The static ArrayList in ContactService meant that every test shared state with every other test. Tests passed in isolation and failed in sequence — a bug that only appears at integration time is one of the hardest categories to diagnose, and it was caused by a single keyword (static) in the wrong place.

Cleaning up the class structure — separating what a Contact is from where Contacts live from how Contacts are accessed — made the underlying logic obvious in a way that reading the original code did not. The Appointment class had methods that looked functional but silently returned null; only tests that checked actual return values revealed them.

The re-setup process (redownloading Eclipse, reconciling a mismatched Maven folder structure from GitHub) also reinforced that getting a project to build again after time away is a real skill — and one that matters in professional environments where you are not always working on code you wrote.
