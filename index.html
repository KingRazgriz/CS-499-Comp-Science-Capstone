Professional Self-Assessment

Who I Am & Where I'm Going

Completing the Computer Science program at SNHU has been more than an academic exercise — it has been a structured process of building the habits, instincts, and technical foundations that define a professional software developer. I entered the program with curiosity and leave it with a clearer identity: a software developer who values honest craftsmanship, systematic debugging, and the discipline to revisit earlier work with fresh eyes.

The three artifacts in this ePortfolio tell that story directly. My CS-330 OpenGL project demonstrates software design and engineering — rebuilding a 3D scene from scratch, resolving memory leaks, implementing three-point lighting, and engineering transparency effects for a wine glass and candle. My CS-320 Java service system demonstrates algorithms and data structures — replacing inefficient ArrayList lookups with HashMap-backed services and introducing real search, sort, and filter capabilities that make the system genuinely useful. My CS-340 MongoDB dashboard demonstrates databases — moving from static CSV files to a live Atlas cluster, designing compound indexes matched to real query patterns, and building server-side aggregation pipelines that shift computation where it belongs.

Across all three, I encountered the same recurring lesson: understanding how the pieces of a system connect is just as important as the logic written within them. Dependency mismatches in Python broke a working dashboard. A misplaced static field in Java caused tests to corrupt each other. An OpenGL initialization order caused null function pointers. Each failure forced me to slow down, trace the system, and reason carefully — exactly the skills that matter in production environments.

My concentration in Software Development has shaped how I approach problems. I think about modularity first — code should be written so that a new team member can extend one piece without breaking another. I think about trade-offs honestly — a fast database query does not mean a fast application if the frontend can't handle the data volume. And I think about maintenance as a first-class concern — dependencies that are correct today become bugs tomorrow, and the ability to re-enter an old codebase and make it run again is a skill in itself.

Looking forward, I am particularly attentive to the rise of generative AI as a collaborative development tool and edge computing as a deployment reality. Both reinforce the same conclusion: strong fundamentals — data structures, system thinking, clear communication of code intent — become more valuable as tooling accelerates, not less. This portfolio is my evidence that those fundamentals are in place.

Software Design & Engineering
CS-330

3D Scene — OpenGL Rendering Engine

A fully modeled 3D scene built with OpenGL 4.6, C++17, GLFW, GLEW, and GLM. Enhanced from a base kitchen layout into a Skyrim-inspired tavern scene featuring a five-part wine glass with transparent wine, a multi-part candle with a glowing floor effect, three-point lighting, and a free-flying camera with scroll-wheel zoom.

  • Fixed critical memory leak (glGenTextures → glDeleteTextures)
  • Implemented alpha blending with correct render order for transparency

  • Shader/texture validation with descriptive error messages

CS-340

Grazioso Salvare Animal Rescue Dashboard

A Python/Dash dashboard backed by MongoDB, originally using basic find() queries against a static CSV. Enhanced to connect to a live MongoDB Atlas cluster with a CRUD module featuring compound indexes, server-side aggregation pipelines for statistics, and a dynamic multi-filter search that assembles queries at runtime.
