Grazioso Salvare Animal Rescue Dashboard

A Python/Dash dashboard backed by MongoDB, enhanced from basic CSV-backed find() queries to a live Atlas cluster with compound indexes, server-side aggregation pipelines, and a dynamic multi-filter search system.

Download Original Download Enhanced

Artifact Description

This artifact is the Grazioso Salvare Animal Rescue Dashboard, originally developed for CS-340: Client/Server Development. It is a Python application using the Dash web framework, running in Jupyter notebooks, with MongoDB as its data layer. The original version used basic find() queries against an in-memory pandas DataFrame loaded from a static dataset. The dashboard allowed users to filter rescue animals by breed and view results on an interactive geo-map.

For this enhancement, I rebuilt the MongoDB CRUD module and migrated the entire application from a static CSV to a live MongoDB Atlas cluster, then designed and implemented indexes, aggregation pipelines, and a dynamic multi-filter search system.

Why I Selected This Artifact

This project was selected because it provides a realistic path from beginner database usage — basic CRUD, no indexing, no server-side computation — to professional database design. The progression from find() queries with pandas post-processing to compound indexes and aggregation pipelines is the same progression that appears in real production systems when query performance starts to matter.

It also gave me the opportunity to work in a live environment rather than a sandboxed one, provisioning a real Atlas cluster, creating users, configuring network access, and troubleshooting authentication — all skills that don't appear in coursework but appear on day one of a real job.

Enhancements Implemented

  • create_indexes(): single-field indexes on animal type, breed, sex, and age; compound index on animal type + breed matching the dashboard's most common query pattern
  • get_breed_statistics(): aggregation pipeline computing breed counts and average ages server-side, replacing equivalent pandas operations in Python
  • get_rescue_summary(): aggregation pipeline summarizing outcomes by rescue category, computed in the database
  • advanced_search(): dynamically assembles a single aggregation pipeline from runtime filter state — whichever combination of animal type, breed, sex, and age range the user selects becomes one pipeline, not chained separate queries
  • Migrated from jupyter_dash (incompatible with current Dash) to the current API: Dash(__name__) and app.run(jupyter_mode=...)
  • Connected dashboard to live MongoDB Atlas cluster with proper credential management and network access configuration
  • Verified each new CRUD method independently against the live collection before wiring into Dash callbacks

Course Outcomes Addressed

Algorithmic Solutions

Index design was reasoned against real query patterns. Compound vs. single-field index choice reflects genuine trade-off analysis, not defaults.

Managing Trade-offs

Testing against the full 10,000-record dataset revealed that fast database queries did not mean a fast application — browser-side data table rendering was the bottleneck. A 500-record working subset was used and documented.

Well-Founded Techniques

Using MongoDB's aggregation framework instead of Python pandas reflects the industry-standard principle: compute where the data lives.

Innovative Tools

Dynamic pipeline assembly in advanced_search() uses MongoDB's composable aggregation model as a first-class design tool, not an afterthought.

Reflection

The most valuable outcome of this enhancement was learning what happens when two parts of a system that work independently fail together. The MongoDB queries were fast. The aggregation pipelines were correct. And the dashboard was still slow because the Dash data table couldn't render 10,000 rows in the browser without stalling. A fast database does not mean a fast application — that lesson, encountered concretely rather than theoretically, is one I will not forget.

The dependency migration also reinforced something important about maintenance. jupyter_dash was correct when the original project was built. A year later it produced a cryptic AttributeError inside a third-party library, with no obvious connection to the actual problem. Debugging it required understanding what jupyter_dash was doing and why the current Dash version no longer needed it. That kind of reasoning — about system evolution over time, not just the system as it exists today — is what distinguishes maintenance-capable engineers.

The process of provisioning a real Atlas cluster, creating database users, configuring IP access lists, and debugging authentication from scratch also gave me hands-on experience with the operational side of database management that coursework environments abstract away entirely.
